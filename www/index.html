Loom

A code-and-debug toolkit for agentic development.

A federation of separately-installable developer tools. Every tool is fully useful on its own; tools compose pairwise to add capability, and removing one never breaks another. SQLite-backed, no cloud, no accounts — and MCP-first, so AI coding agents use them as first-class citizens alongside a full CLI for humans.

The 1.0 toolkit is four tools spanning the build→debug→ship loop. Two more are joining.

The architectural promise #

Loom is a federation, not a monolith. Three rules hold it together — and keep any single tool honest about working alone.

Solo-useful

Every tool has a complete, respectable use case by itself. Zero siblings required — you can install one and get full value with nothing else present.

Enrich-only, never load-bearing

Tools compose pairwise to add capability, but a sibling’s absence only reduces convenience — it never changes the meaning of your data or stops a tool from booting.

Local-first & agent-native

Local SQLite storage, no cloud, no accounts. Every tool is MCP-first so coding agents drive it directly, with a full CLI for humans.

Loom 1.0 — the code-and-debug toolkit #

Four tools, read as a loop: track the work, understand the code, find the weaknesses, govern the change. Each installs separately and runs on its own.

filigree

Python

Track the work

Local-first, agent-native issue tracker. Holds work state — issues, plans, observations, findings — in SQLite, exposed over MCP tools. No cloud.

pip install filigree

Code archaeology. Extracts a structural entity graph from a codebase and serves graph-aware briefings to LLM agents over MCP. The suite’s identity authority.

How they compose #

The four tools read as one loop. Each step is independently useful; the arrows are enrichment, not dependency.

  1. 1

    Track

    filigree holds the work — what exists, what state it’s in, what happened.

  2. →
  3. 2

    Understand

    clarion maps the codebase into a structural graph agents can reason over.

  4. →
  5. 3

    Probe

    wardline traces taint across trust boundaries and emits findings.

  6. →
  7. 4

    Govern

    legis grades the change and records SEI-keyed attestations before it ships.

One durable identity underneath

Each tool models a codebase as entities carrying typed facts — an issue here, a structural fact there, a taint verdict, a governance sign-off — all keyed on one shared, durable identity. That common operating picture is what lets a finding in one tool line up with an entity in another, without any tool becoming the system of record for the rest.

Roadmap — joining the weave #

Two tools extend Loom beyond the code-and-debug core: charter reaches up into requirements, shuttle reaches down into change execution — bookending the four-tool loop. Neither is part of 1.0.

charter

Python In progress

Upstream of code — “what must be true”

Requirements, traceability, baselines, and verification evidence. The authority for the obligations a codebase has to satisfy, sitting upstream of the build loop.

Not yet 1.0

shuttle

Future Sketched

Downstream of code — applying approved change

Transactional, scoped change execution under guardrails: carry an approved change through incrementally with pre/post checks and rollback. A sketched future authority — no repo yet.
